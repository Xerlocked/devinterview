# 게임 개발자 C++ & 운영체제 30문항

## [PART 1] C++ Core & Memory Management

1. 가상 소멸자(Virtual Destructor): 상속 관계에서 부모 클래스의 소멸자를 virtual로 선언해야 하는 이유와, 선언하지 않았을 때 발생하는 문제를 메모리 관점에서 설명하세요.
<!-- answer-start -->
부모 클래스의 포인터로 자식 객체를 참조할 때, `delete`를 호출하면 부모의 소멸자가 호출됩니다. 이때 부모 소멸자가 `virtual`이 아니면 부모 소멸자만 호출되고 자식 소멸자는 호출되지 않아 자식 객체가 할당한 메모리가 누수(Memory Leak)됩니다. `virtual`로 선언하면 vtable을 통해 실제 자식 객체의 소멸자가 먼저 호출되어 메모리가 안전하게 해제됩니다.
<!-- answer-end -->

2. RAII (Resource Acquisition Is Initialization): C++에서 자원을 관리하는 이 설계 패턴의 핵심 원리와, 이것이 예외 처리(Exception Safety)에 어떻게 기여하는지 서술하세요.
<!-- answer-start -->
RAII는 객체의 생성자에서 자원을 획득하고 소멸자에서 자원을 해제하는 기법입니다. C++은 스코프를 벗어날 때 스택에 생성된 지역 객체의 소멸자를 자동으로 호출하는 특성이 있습니다. 따라서 중간에 예외(Exception)가 발생해 함수를 빠져나가더라도 스택 풀림(Stack Unwinding) 과정에서 소멸자가 보장되어 안전하게 자원(메모리, 뮤텍스 등)이 해제됩니다.
<!-- answer-end -->

3. 스마트 포인터(Smart Pointers): unique_ptr, shared_ptr, weak_ptr의 차이점을 설명하고, 특히 weak_ptr이 '순환 참조(Circular Reference)' 문제를 어떻게 해결하는지 설명하세요.
<!-- answer-start -->
`unique_ptr`은 단독 소유권을 가지며 복사할 수 없고 이동만 가능합니다. `shared_ptr`은 참조 카운팅을 통해 여러 곳에서 공유할 수 있으며 카운트가 0이 될 때 메모리를 해제합니다.
`weak_ptr`은 `shared_ptr`을 관찰하지만 참조 카운트를 증가시키지 않습니다. 두 객체가 `shared_ptr`로 서로를 가리키면 카운트가 0이 되지 않는 순환 참조가 발생하는데, 한쪽을 `weak_ptr`로 선언하여 참조 카운트를 올리지 않음으로써 이를 해결합니다.
<!-- answer-end -->

4. L-value vs R-value: C++11 이후 도입된 우측값 참조(R-value Reference)의 개념과, 이를 활용한 이동 생성자(Move Constructor)가 성능 최적화에 미치는 영향은 무엇인가요?
<!-- answer-start -->
L-value는 메모리 주소가 있어 이름을 통해 접근할 수 있는 값이고, R-value는 연산 직후 사라지는 임시 객체입니다. 우측값 참조(`&&`)는 이 임시 객체를 식별합니다. 
이동 생성자는 R-value 참조를 통해 원본의 데이터를 깊은 복사(Deep Copy)하는 대신 포인터만 가로채는(소유권 이전) 얕은 복사를 수행합니다. 이는 임시 객체의 복사 오버헤드를 극적으로 줄여 성능을 최적화합니다.
<!-- answer-end -->

5. 가상 함수 테이블(vtable): C++에서 다형성이 구현되는 원리인 vptr과 vtable의 동작 방식을 설명하세요.
<!-- 무answer-start -->
클래스에 하나 이상의 가상 함수가 있으면 컴파일러는 가상 함수 포인터들을 담은 배열인 **vtable(가상 함수 테이블)**을 만듭니다. 그리고 해당 클래스의 각 객체 내부에 vtable을 가리키는 숨겨진 포인터인 **vptr**을 추가합니다. 
런타임에 가상 함수가 호출되면, 객체의 vptr을 통해 해당 클래스의 vtable에 접근하여 실제 호출해야 할 함수의 주소를 찾아 실행하는 동적 바인딩(Dynamic Binding)이 일어납니다.
<!-- answer-end -->

6. 깊은 복사 vs 얕은 복사: 클래스 내부에 동적 할당된 포인터가 있을 때, 사용자 정의 복사 생성자가 필요한 이유를 설명하세요.
<!-- answer-start -->
컴파일러가 제공하는 기본 복사 생성자는 멤버 변수의 값만 그대로 복사하는 **얕은 복사(Shallow Copy)**를 수행합니다.
포인터 멤버가 있을 경우 주소값만 복사되므로, 두 객체가 같은 동적 메모리를 공유하게 됩니다. 하나의 객체가 소멸할 때 메모리를 해제하면, 다른 객체는 해제된 메모리를 가리키는 댕글링 포인터(Dangling Pointer)가 되어 충돌이 발생합니다. 따라서 동적 할당을 직접 새로 하여 데이터를 복사하는 **깊은 복사**를 구현해야 합니다.
<!-- answer-end -->

7. 캐스팅(Casting): static_cast, dynamic_cast, reinterpret_cast, const_cast의 용도와 차이점을 설명하세요.
<!-- answer-start -->
- `static_cast`: 기본 자료형 변환이나, 논리적으로 변환 가능한 타입(업캐스팅/다운캐스팅) 간에 컴파일 타임 변환을 수행합니다.
- `dynamic_cast`: 다형성을 가진(가상 함수가 있는) 부모-자식 간에 런타임 타입 검사를 포함하여 다운캐스팅합니다. 실패 시 `nullptr`을 반환합니다.
- `reinterpret_cast`: 포인터 타입을 완전히 다른 포인터 타입이나 정수로 강제 변환합니다. 가장 위험합니다.
- `const_cast`: 포인터나 참조자의 `const` 속성을 제거할 때 사용합니다.
<!-- answer-end -->

8. 템플릿(Template): 템플릿 특수화(Template Specialization)란 무엇이며, 컴파일 타임에 코드가 생성되는 방식이 런타임 성능에 어떤 이점을 주는지 서술하세요.
<!-- answer-start -->
템플릿 특수화는 제네릭한 템플릿 코드와 별개로, 특정 데이터 타입에 대해 맞춤형 구현을 별도로 정의하는 기법입니다.
템플릿은 컴파일 타임에 사용된 타입에 맞춰 실제 함수나 클래스 코드를 완전히 독립적으로 인스턴스화하여 생성합니다. 이로 인해 런타임에 타입 체크나 동적 바인딩을 할 필요가 없어져 오버헤드가 발생하지 않으므로, 성능 손실 없이 제네릭 프로그래밍을 할 수 있습니다.
<!-- answer-end -->

9. 메모리 정렬(Memory Alignment): 구조체나 클래스에서 멤버 변수의 순서에 따라 전체 크기가 달라지는 이유(Padding)와 성능과의 관계를 설명하세요.
<!-- answer-start -->
CPU는 메모리를 읽을 때 4바이트, 8바이트 단위(워드 단위)로 읽는 것이 가장 빠릅니다. 컴파일러는 이 정렬 요구 조건(Alignment)을 맞추기 위해 변수들 사이에 의미 없는 빈 공간(Padding Byte)을 삽입합니다.
따라서 크기가 큰 변수부터 내림차순으로 선언하면 패딩을 최소화하여 메모리를 아낄 수 있습니다. 정렬을 맞추면 CPU가 한 번의 메모리 접근으로 데이터를 읽을 수 있어 캐시 효율과 접근 속도가 향상됩니다.
<!-- answer-end -->

10. Inline 함수: inline 키워드가 컴파일러에게 주는 힌트와, 함수 호출 오버헤드를 줄이는 대신 발생할 수 있는 잠재적 단점(Code Bloat)을 설명하세요.
<!-- answer-start -->
`inline` 키워드는 함수 호출부를 실제 함수의 코드로 치환해 달라고 컴파일러에 요청(힌트)합니다. 이를 통해 함수 호출 시 발생하는 스택 프레임 생성, 레지스터 저장 등의 오버헤드를 제거할 수 있습니다.
하지만 코드가 치환되어 삽입되므로, 함수의 크기가 크거나 여러 곳에서 빈번하게 호출되면 실행 파일의 크기가 비대해지는 **코드 블로트(Code Bloat)**가 발생할 수 있으며, 이는 오히려 캐시 히트율을 떨어뜨려 성능 저하를 유발할 수 있습니다.
<!-- answer-end -->

## [PART 2] Low-level & Optimization

11. Stack vs Heap: 메모리 영역으로서의 스택과 힙의 차이(할당 속도, 수명 관리, 크기 제한)를 비교 설명하세요.
<!-- answer-start -->
- **할당 속도**: 스택은 포인터(Stack Pointer)만 이동시키므로 매우 빠릅니다. 반면 힙은 빈 공간을 탐색하고 관리 메타데이터를 갱신해야 하므로 상대적으로 느립니다.
- **수명 관리**: 스택은 스코프를 벗어나면 자동으로 반환되지만, 힙은 프로그래머가 직접 `new/delete`로 관리해야 합니다.
- **크기 제한**: 스택은 OS에 의해 비교적 작게(보통 몇 MB) 고정되어 있어 재귀가 깊어지면 스택 오버플로우가 발생합니다. 힙은 물리적 RAM과 가상 메모리가 허용하는 한 넉넉하게 사용 가능합니다.
<!-- answer-end -->

12. 포인터 vs 참조자(Reference): 두 개념의 차이점을 문법적, 메모리적 관점에서 비교하고 어떤 상황에 참조자를 쓰는 것이 권장되는지 설명하세요.
<!-- answer-start -->
포인터는 변수의 주소를 저장하는 독립된 변수로 `nullptr` 할당 및 재할당이 가능합니다. 참조자는 기존 변수에 대한 별칭(Alias)으로 선언 시 반드시 초기화해야 하며 `nullptr`을 가질 수 없고 다른 대상을 가리키도록 재할당할 수 없습니다.
내부적으로 참조자는 상수로 취급되는 포인터로 구현됩니다. 값이 반드시 존재해야 하고 변경되지 않는 대상을 참조할 때는 **참조자**를 사용하는 것이 널 포인터 검사를 생략할 수 있어 안전하고 직관적입니다.
<!-- answer-end -->

13. const의 활용: 함수의 파라미터나 리턴 타입, 멤버 함수 뒤에 붙는 const가 각각 어떤 의미를 가지며 프로그래머의 실수를 어떻게 방지하는지 서술하세요.
<!-- answer-start -->
- **파라미터 const**: 넘어온 변수(특히 참조자)의 값을 함수 내부에서 수정하지 않겠다는 보장입니다.
- **리턴 타입 const**: 반환된 객체의 값을 호출자가 수정하지 못하게 막습니다.
- **멤버 함수 뒤 const**: 해당 객체의 멤버 변수를 변경하지 않는 읽기 전용 함수임을 명시합니다. (const 객체는 이 함수만 호출 가능)
이를 통해 의도치 않은 값 변경으로 인한 버그를 컴파일 타임에 원천 차단하여 안전성을 높입니다.
<!-- answer-end -->

14. STL 컨테이너 최적화: std::vector와 std::list의 메모리 구조 차이를 설명하고, 왜 현대 CPU에서는 연속된 메모리 구조(vector)가 성능상 유리한지 캐시 지역성(Cache Locality) 관점에서 설명하세요.
<!-- answer-start -->
`std::vector`는 메모리가 연속적으로 할당된 배열 구조이고, `std::list`는 노드들이 메모리 파편화되어 링크로 연결된 구조입니다.
현대 CPU는 메모리에서 데이터를 가져올 때 캐시 라인 단위(보통 64바이트)로 뭉텅이로 가져옵니다. `vector`는 인접한 데이터가 이미 캐시에 로드되어 있을 확률이 높아 **공간적 지역성(Spatial Locality)**을 잘 활용해 캐시 히트율이 높습니다. 반면 `list`는 메모리가 흩어져 있어 매번 캐시 미스가 발생하므로 탐색 성능이 매우 떨어집니다.
<!-- answer-end -->

15. 이름 공간(Namespace): 대규모 게임 프로젝트에서 이름 공간을 사용하는 목적과 using namespace std;를 헤더 파일에서 지양해야 하는 이유를 설명하세요.
<!-- answer-start -->
이름 공간은 수많은 클래스와 함수들을 논리적으로 묶어 **이름 충돌(Name Collision)**을 방지하는 역할을 합니다. 
헤더 파일에 `using namespace std;`를 선언하면, 이 헤더를 포함(include)하는 모든 파일의 전역 네임스페이스가 `std`에 오염됩니다. 이로 인해 내가 정의한 함수/변수와 STL의 이름(예: `count`, `max`)이 예기치 않게 충돌하여 원인을 찾기 힘든 컴파일 에러를 유발할 수 있으므로 절대 지양해야 합니다.
<!-- answer-end -->

16. 함수 객체(Functor) vs 람다(Lambda): 람다 표현식이 내부적으로 어떻게 동작(클로저 생성)하는지 설명하고, std::sort 등에서 활용되는 사례를 쓰세요.
<!-- answer-start -->
람다 표현식을 컴파일하면 내부적으로 `operator()`가 오버로딩된 이름 없는 익명 클래스(Functor)가 생성되며, 캡처된 외부 변수들은 이 클래스의 멤버 변수로 저장(클로저 객체 생성)됩니다.
`std::sort`의 커스텀 정렬 조건으로 자주 사용됩니다. 예: `std::sort(v.begin(), v.end(), [](int a, int b) { return a > b; });` 이처럼 별도의 함수나 클래스를 선언하지 않고도 간결하게 콜백 로직을 작성할 수 있습니다.
<!-- answer-end -->

17. RTTI (Run-Time Type Information): 런타임에 객체의 타입을 식별하는 원리와, 게임 엔진(예: Unreal)에서 이를 최적화하기 위해 자체적으로 구현하는 방식에 대해 고찰하세요.
<!-- answer-start -->
RTTI는 `dynamic_cast`나 `typeid`를 위해 vtable 포인터 근처에 해당 타입의 정보 구조체(type_info) 포인터를 저장하여 런타임에 타입을 식별합니다.
이는 성능 오버헤드가 있고 메모리를 차지하기 때문에 언리얼 엔진 같은 곳에서는 네이티브 RTTI를 끄고 자체 리플렉션 시스템(UObject, StaticClass())을 사용합니다. 각 객체에 정수형 타입 ID를 부여하거나 안전한 캐스팅(`Cast<T>`) 매크로를 구현하여 훨씬 빠르고 가볍게 타입 식별을 처리합니다.
<!-- answer-end -->

18. std::move와 std::forward: std::move가 실제로 데이터를 옮기는지 여부와, 완전한 전달(Perfect Forwarding)에서 std::forward의 역할을 설명하세요.
<!-- answer-start -->
`std::move`는 데이터를 직접 옮기지 않고, 객체를 단순히 무조건 우측값 참조(`T&&`)로 캐스팅하여 이동 생성자/대입 연산자가 호출될 수 있도록 만들어줍니다.
`std::forward`는 템플릿 프로그래밍에서 넘겨받은 인자의 원본 타입이 L-value인지 R-value인지 기억했다가, 그 속성을 그대로 유지한 채 다른 함수로 전달(Perfect Forwarding)하기 위해 사용됩니다. (주로 보편적 참조 `T&&`와 함께 쓰임)
<!-- answer-end -->

19. 메모리 릭(Memory Leak): 메모리 누수를 방지하기 위한 C++의 도구들(valgrind, CRT 라이브러리 등)이나 습관에 대해 서술하세요.
<!-- answer-start -->
가장 좋은 습관은 `new`/`delete`를 직접(Raw Pointer) 사용하지 않고 **스마트 포인터**(`std::unique_ptr` 등)를 적극 사용하여 RAII 패턴으로 관리하는 것입니다.
도구로는 리눅스 환경의 `Valgrind`, 윈도우 환경의 `_CrtDumpMemoryLeaks()` 함수, 또는 Visual Studio의 진단 도구가 있습니다. 게임 엔진들은 자체적인 커스텀 메모리 할당자(Custom Allocator)를 구현하여 할당/해제 횟수와 주소를 트래킹하여 메모리 누수를 감지합니다.
<!-- answer-end -->

20. 예외 처리(Exception Handling): try-catch 블록이 실행될 때 발생하는 성능 비용과, 게임 엔진의 핵심 루프에서 예외 처리를 지양하는 이유를 설명하세요.
<!-- answer-start -->
try-catch는 컴파일러가 스택 풀림(Stack Unwinding)을 위해 예외 처리 테이블과 실행 경로 메타데이터를 유지해야 하므로, 실행 파일이 커지고 정상적인 실행 경로에서도 미세한 오버헤드가 발생합니다. 
특히 예외가 던져질 때(throw) 수많은 스택 프레임을 거슬러 올라가며 객체를 해제하는 비용은 매우 큽니다. 매 프레임이 중요한 게임의 핵심 렌더/업데이트 루프에서는 이러한 불규칙한 성능 저하(히치 현상)를 방지하기 위해 예외 대신 반환값(오류 코드, `std::optional` 등)을 사용합니다.
<!-- answer-end -->

## [PART 3] 운영체제 및 멀티스레딩 (OS & Concurrency)

21. 프로세스 vs 스레드: 운영체제 관점에서 두 개념의 정의와, 자원 공유 방식(메모리 공간)의 차이를 설명하세요.
<!-- answer-start -->
프로세스는 실행 중인 프로그램의 인스턴스로, OS로부터 독립적인 메모리 공간(Code, Data, Heap, Stack)을 할당받습니다. 스레드는 프로세스 내부의 실행 흐름 단위입니다.
프로세스 간에는 메모리가 격리되어 통신(IPC)이 복잡하지만, 한 프로세스 내의 스레드들은 Code, Data, Heap 영역을 공유하고 자신만의 Stack만 따로 갖습니다. 이로 인해 스레드는 생성과 통신이 빠르지만 동기화 문제(경쟁 상태)에 취약합니다.
<!-- answer-end -->

22. 컨텍스트 스위칭(Context Switching): 스레드 전환 시 CPU에서 어떤 일이 일어나는지 설명하고, 스레드가 너무 많을 때 시스템 성능이 저하되는 이유를 서술하세요.
<!-- answer-start -->
컨텍스트 스위칭은 실행 중인 스레드의 상태(레지스터, PC 등)를 TCB에 저장하고, 다음에 실행할 스레드의 상태를 복원하여 CPU가 실행을 이어가는 과정입니다.
이 전환 과정 중에는 유용한 작업이 수행되지 않는 오버헤드가 발생합니다. 특히 스레드가 너무 많으면 CPU 캐시가 계속 무효화(Cache Flush)되어 캐시 적중률이 급격히 떨어지며, 전환 비용이 실제 연산 시간보다 커져 전체 시스템의 처리량이 저하(Thrashing)됩니다.
<!-- answer-end -->

23. 데드락(Deadlock): 교착 상태가 발생하는 4가지 필수 조건과, 이를 예방하거나 회피하기 위한 전략을 한 가지만 설명하세요.
<!-- answer-start -->
데드락의 4조건은 1)상호 배제, 2)점유 대기, 3)비선점, 4)순환 대기입니다. 
**해결 전략 (순환 대기 예방):** 모든 자원에 고유한 번호(순서)를 부여하고, 스레드는 반드시 번호가 낮은 자원부터 오름차순으로만 자원(뮤텍스)을 요청하도록 락(Lock) 획득 순서를 강제합니다. 이렇게 하면 서로 꼬리를 무는 순환 대기가 원천적으로 발생할 수 없어 데드락을 예방할 수 있습니다.
<!-- answer-end -->

24. 뮤텍스(Mutex) vs 세마포어(Semaphore): 두 동기화 객체의 차이점과, '소유권'의 개념이 어디에 적용되는지 설명하세요.
<!-- answer-start -->
뮤텍스는 오직 1개의 스레드만 임계 구역에 들어갈 수 있게 하는 상호 배제 잠금 장치이며, **소유권** 개념이 있어서 락을 획득한 스레드만이 락을 해제할 수 있습니다.
반면 세마포어는 정수형 카운터를 기반으로 여러 스레드(N개)가 동시에 자원에 접근할 수 있게 허용하는 신호 메커니즘입니다. 소유권 개념이 없으므로, A 스레드가 Wait(감소)한 세마포어를 B 스레드가 Signal(증가)하여 풀어줄 수도 있습니다.
<!-- answer-end -->

25. 경쟁 상태(Race Condition): 여러 스레드가 동시에 같은 메모리에 접근할 때 발생하는 문제와, 이를 방지하기 위한 원자적 연산(Atomic Operation)의 장점을 설명하세요.
<!-- answer-start -->
경쟁 상태는 둘 이상의 스레드가 공유 데이터를 동시에 읽고 쓸 때 타이밍에 따라 최종 결과가 의도치 않게 꼬이는(데이터 오염) 현상입니다.
원자적 연산(`std::atomic`)은 CPU 명령어 레벨에서 해당 연산이 중간에 끊기지 않고 한 번에 끝남을 보장합니다. 뮤텍스를 사용한 락(Lock) 기반 동기화는 컨텍스트 스위칭 오버헤드나 데드락 위험이 있지만, 원자적 연산은 락 없이(Lock-free) 하드웨어적 지원으로 동기화를 매우 빠르게 처리할 수 있습니다.
<!-- answer-end -->

26. 가상 메모리(Virtual Memory): 물리 메모리보다 큰 프로그램을 실행할 수 있게 하는 원리와, 페이지 폴트(Page Fault)가 발생했을 때의 처리 과정을 설명하세요.
<!-- 조answer-start -->
가상 메모리는 프로세스에게 연속되고 독립적인 논리적 주소 공간을 제공하며, 현재 실행에 필요한 일부 페이지(Page)만 물리 메모리(RAM)에 로드하고 나머지는 디스크(Swap 영역)에 둡니다.
CPU가 접근하려는 가상 주소의 페이지가 물리 메모리에 없으면 **페이지 폴트**가 발생합니다. OS는 CPU를 일시 정지시키고, 디스크에서 해당 페이지를 찾아 빈 물리 메모리 프레임에 로드한 뒤 페이지 테이블을 갱신하고 중단된 명령어를 재개합니다.
<!-- answer-end -->

27. 메모리 단편화(Fragmentation): 외부 단편화와 내부 단편화의 차이를 설명하고, 게임 엔진에서 이를 해결하기 위한 '풀링(Pooling)' 기법의 원리를 서술하세요.
<!-- answer-start -->
- **내부 단편화**: OS가 할당한 메모리 블록이 요청 크기보다 커서 내부에 남는 잉여 공간.
- **외부 단편화**: 메모리의 빈 공간들이 파편화되어 흩어져, 총 빈 공간은 충분하지만 연속된 큰 덩어리를 할당하지 못하는 현상.
**오브젝트 풀링(Object Pooling)**은 사용할 객체들을 미리 한 번에 큰 덩어리의 배열로 미리 할당해 두고, 필요할 때 배열에서 꺼내 쓰고 반납하는 기법입니다. 동적 할당(`new`/`delete`)을 피하여 파편화를 막고 생성 비용을 없앱니다.
<!-- answer-end -->

28. CPU 스케줄링: 시분할 시스템에서 우선순위 스케줄링이 게임 프로세스의 반응성에 미치는 영향을 설명하세요.
<!-- answer-start -->
시분할 시스템은 여러 프로세스에게 CPU 시간을 잘게 쪼개어 번갈아 할당(Time Quantum)하여 동시에 실행되는 것처럼 보이게 합니다. 
게임은 60프레임 등 실시간 렌더링과 입력 처리가 중요하므로 반응성이 생명입니다. 게임 프로세스나 핵심 스레드(렌더 스레드, 오디오 스레드)의 스케줄링 **우선순위를 높이면**, OS가 게임에게 CPU 점유 기회를 먼저 부여하여 프레임 드랍(Stuttering)을 막고 입력 지연을 최소화할 수 있습니다. 단, 기아 상태(Starvation)를 주의해야 합니다.
<!-- answer-end -->

29. 캐시 계층 구조 (L1, L2, L3): CPU 캐시의 존재 이유와, 데이터 지향 설계(Data-Oriented Design)가 왜 게임 엔진 최적화의 트렌드인지 캐시 히트(Cache Hit)와 관련지어 설명하세요.
<!-- answer-start -->
CPU 속도는 매우 빠르지만 RAM 속도가 느려 병목이 발생합니다. 이를 완화하기 위해 CPU 곁에 L1, L2, L3라는 작고 빠른 SRAM 캐시 계층을 둡니다.
객체 지향 설계(OOD)는 데이터가 흩어져 있어 캐시 미스가 잦습니다. 반면 **데이터 지향 설계(DOD)**는 자주 함께 처리되는 데이터를 구조체의 배열(SoA) 등 연속된 메모리로 모아 배치합니다. CPU가 메모리를 가져올 때 인접한 데이터를 한 번에 캐시 라인으로 가져오므로, **캐시 히트율이 극대화**되어 수만 개의 엔티티 연산 시 압도적인 성능 향상을 보입니다.
<!-- answer-end -->

30. 유저 모드 vs 커널 모드: 시스템 호출(System Call)이 일어날 때 모드 전환이 발생하는 이유와, 잦은 시스템 호출이 게임 프레임에 미치는 영향을 설명하세요.
<!-- answer-start -->
OS는 시스템을 보호하기 위해 파일, 네트워크, 메모리 등 하드웨어 자원 접근을 권한이 높은 **커널 모드**에서만 가능하도록 제한합니다. 일반 프로그램은 **유저 모드**에서 돌다가 자원이 필요하면 OS에게 시스템 호출(System Call)을 요청하며 이때 모드 전환이 발생합니다.
모드 전환 시 보안 검사와 레지스터 백업 등 큰 오버헤드가 발생합니다. 파일 I/O나 잦은 메모리 동적 할당, 로우레벨 렌더링 호출을 너무 자주 하면 이 전환 오버헤드가 누적되어 게임 프레임 저하를 유발합니다.
<!-- answer-end -->
